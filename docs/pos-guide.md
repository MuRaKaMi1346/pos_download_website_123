# SmartBrew POS — คู่มือใช้งาน & การตั้งค่าเริ่มต้น

> คู่มือสำหรับแอดมินและแคชเชียร์ ครอบคลุมหน้าจอ POS ที่สร้างใน milestone 12–19
> (pro-pos-upgrade). อ้างอิงสเปกเต็มที่ [`architecture-spec.md`](./architecture-spec.md)
> และ prompt ที่ [`prompts/pro-pos-upgrade.prompt.md`](./prompts/pro-pos-upgrade.prompt.md).

---

## 1. ภาพรวม

SmartBrew POS เป็นระบบขายหน้าร้าน (touch-first) สำหรับร้านกาแฟ — เลือกเมนู →
ปรับตัวเลือก → ชำระเงินหลายช่องทาง → พิมพ์ใบเสร็จ พร้อมระบบกะเงินสด, ลูกค้า/แต้มสะสม,
บาร์โค้ด และจอครัว (KDS)

**เส้นทาง (routes)** ทั้งหมดต้องล็อกอินก่อน:

| Route | หน้าจอ | สิทธิ์ |
| --- | --- | --- |
| `/login` | เข้าสู่ระบบ | — |
| `/pos` | หน้าขายหลัก (3 คอลัมน์) | staff/admin · **ต้องเปิดกะ** |
| `/shift` | เปิด/ปิดกะเงินสด | staff/admin |
| `/kds` | จอครัว (BAR/KITCHEN) | staff/admin |
| `/dashboard` | แดชบอร์ดยอดขาย | admin |
| `/ai-insights` | คำแนะนำ AI | admin |

> หมายเหตุ: การจัดการ **แคตตาล็อก (สินค้า/หมวดหมู่/ตัวเลือก)** และ **การตั้งค่าร้าน**
> ทำผ่าน **API** (ยังไม่มีหน้าจอแอดมินใน frontend) — ดู §6

## 2. เริ่มต้นใช้งาน (Admin Onboarding)

### 2.1 รันเซิร์ฟเวอร์

```bash
# Backend (จากโฟลเดอร์ backend/)
uv sync
uv run alembic upgrade head          # สร้าง/อัปเดตสคีมา
uv run uvicorn app.main:app --reload --port 8000

# Frontend (จากโฟลเดอร์ frontend/)
pnpm install
pnpm dev                              # http://localhost:5173
```

API docs (Swagger) อยู่ที่ **http://localhost:8000/docs** — ใช้ทดลองยิง API ทุกตัวได้

### 2.2 เข้าสู่ระบบครั้งแรก

ผู้ใช้แอดมินถูก seed อัตโนมัติตอน backend เริ่มทำงาน (ถ้ายังไม่มี) ค่าเริ่มต้น
(เมื่อไม่มีไฟล์ `backend/.env`):

- **username:** `admin`
- **password:** `change-me-on-first-run`

⚠️ ตั้งค่า `SEED_ADMIN_USERNAME` / `SEED_ADMIN_PASSWORD` ใน `backend/.env`
(คัดลอกจาก `.env.example`) **ก่อนรันครั้งแรก** สำหรับ production

### 2.3 ตั้งค่าร้าน (Store Settings)

ผ่าน `PATCH /api/v1/settings` (admin) — คีย์ที่ใช้บ่อย:

| คีย์ | ความหมาย |
| --- | --- |
| `store_name`, `store_address`, `store_tax_id` | หัวใบเสร็จ |
| `vat_rate` (เช่น `0.07`), `tax_inclusive` | VAT และรวม/ไม่รวมในราคา |
| `service_charge_rate`, `service_charge_before_vat` | ค่าบริการ |
| `rounding_mode` (`TWO_DECIMALS` \| `NEAREST_BAHT`) | การปัดเศษบิล |
| `default_channel` | ช่องทางตั้งต้น |
| `loyalty_baht_per_earn_point`, `loyalty_baht_per_redeem_point` | อัตราแต้มสะสม |
| `receipt_footer`, `receipt_pdf_enabled`, `printer_name` | ใบเสร็จ |

ค่าเริ่มต้นมาจาก env/config และถูก override ด้วยค่าใน DB ที่ตั้งผ่าน API

### 2.4 สร้างแคตตาล็อก

ผ่าน admin API (ดู `/docs`):

- **สินค้า** — `POST /api/v1/products/` (`name`, `price`, `cost`, `category_id`,
  `image`, **`sku`**, **`barcode`**) — `sku`/`barcode` ใช้สำหรับสแกน (§4.7)
- **ตัวเลือก (modifiers)** — `POST /api/v1/modifier-groups/` กำหนด
  `min_select`/`max_select`/`is_required` + รายการ modifier (radio เมื่อ
  `max_select=1`, checkbox เมื่อ >1) แล้วผูกกับสินค้า
- **สูตร (recipes)** — ผูกวัตถุดิบกับสินค้า/ตัวเลือกเพื่อตัดสต็อก

> สินค้าจะโชว์จุด "ตัวเลือก" บนการ์ดเมื่อมี modifier ที่ active

## 3. ขั้นตอนการขายประจำวัน (Cashier Workflow)

1. **เปิดกะ** — เข้า `/pos` ครั้งแรกจะถูกพาไป `/shift` ให้กรอก *เงินตั้งต้น*
   (opening float) แล้วกด **เปิดกะ** (กะจะค้างฝั่งเซิร์ฟเวอร์จนกว่าจะปิด)
2. **เลือกช่องทาง** — Dine-in / Takeaway / Delivery บนแถบบนสุด (ใส่เลขโต๊ะถ้า dine-in)
3. **เพิ่มเมนู** — แตะการ์ดเมนู, ค้นหา (`/`), หรือสแกนบาร์โค้ด (`⌘/Ctrl+K`)
   - สินค้าที่มีตัวเลือก → เปิด dialog ให้เลือก (radio/checkbox) + ใส่หมายเหตุ
   - แก้บรรทัดในตะกร้า → แตะบรรทัดเพื่อเปิดแผง (จำนวน/ตัวเลือก/หมายเหตุ/ลบ)
4. **แนบลูกค้า** (ไม่บังคับ) — แตะชิป "Walk-in" → ค้นหา/เพิ่มลูกค้า → ได้แต้มเมื่อชำระเงิน
5. **ชำระเงิน** — กด **ชำระเงิน** (`F9`) → เลือกช่องทาง (เงินสด/พร้อมเพย์/บัตร/อื่น ๆ),
   ใส่ยอด, รองรับหลายช่องทาง (split), ระบบคำนวณ *เงินทอน* → ยืนยัน
6. **ใบเสร็จ** — หน้าจอสำเร็จแสดงตัวอย่างใบเสร็จ 80mm + ปุ่ม **พิมพ์** → **ขายใหม่**
7. **พักบิล** — กด **พักบิล** (`F8`) เพื่อเก็บบิลไว้ก่อน เรียกคืนทีหลังจากปุ่ม "พักบิล" บนแถบบน
8. **ปิดกะ** — เมนู **กะ** บน header → นับเงินสด → ระบบแสดง *คาดหวัง vs นับได้ + ส่วนต่าง*

## 5. คีย์ลัด (POS)

| ปุ่ม | การทำงาน |
| --- | --- |
| `/` | โฟกัสช่องค้นหา |
| `Ctrl` / `⌘` + `K` | ค้นหาเมนู / สแกนบาร์โค้ด |
| `F2` | แนบลูกค้า |
| `F4` | สลับช่องทางการขาย |
| `F8` | พักบิล |
| `F9` | ชำระเงิน |
| `Esc` | ปิดหน้าต่าง / ล้างการค้นหา |

ดูได้จากปุ่มรูปคีย์บอร์ดบนแถบบน · ปุ่มข้าง ๆ สลับ **ความหนาแน่น** (compact/comfortable,
จำค่าไว้ใน browser)

## 6. งานของแอดมิน

- **แคตตาล็อก** — สินค้า/ตัวเลือก/สูตร ผ่าน admin API (`/docs`)
- **ตั้งค่าร้าน** — `GET`/`PATCH /api/v1/settings` (ดู §2.3)
- **ประวัติกะ** — `GET /api/v1/shifts/` (admin) ดูส่วนต่างเงินสดย้อนหลัง
- **Audit log** — `GET /api/v1/audit-logs` (void/refund/discount/settings/shift)
- **แดชบอร์ด & AI** — `/dashboard`, `/ai-insights`

## 7. ส่วนต่างจาก spec + ช่องว่างที่ทราบ (Spec deltas & known gaps)

สิ่งที่ตั้งใจ "เลื่อน" หรือทำต่างจาก prompt — เพื่อให้แต่ละ milestone ปิดงานได้สะอาด:

1. **ยังไม่มี "ส่งเข้าครัว" (send-to-kitchen) ใน frontend** — `POST
   /orders/{id}/send-to-kitchen` มีฝั่ง backend แล้ว แต่ตะกร้ายัง "ชำระเงิน" โดยไม่ส่งเข้าครัว
   ผลคือ **จอ KDS จะว่าง** จนกว่าจะมีการเรียก action นี้ และ **สต็อกไม่ถูกตัด**
   (M3 ตัดสต็อกตอน send-to-kitchen) — ช่องว่างสะสมที่ควรทำเป็นงานแยก
2. **UI ส่วนลด/ทิป ยังไม่ทำ** — discount engine มีฝั่ง backend (M4) แต่ยังไม่มีหน้าจอ
   ใส่คูปอง/ส่วนลดบิล-บรรทัด/ทิป และ **แลกแต้ม (redeem)** ถูกเลื่อน เพราะมันสร้างส่วนลด
   ที่ทำให้ยอดสุทธิเปลี่ยน ซึ่งจะชนกับโฟลว์ชำระเงิน (สร้างบิลตอนกดยืนยัน) ของ M14
3. **หมายเหตุรายบรรทัดไม่ถูกบันทึก** — `OrderItemCreate` ยังไม่รับ note ดังนั้นหมายเหตุ
   เป็น local-only (ไม่รอดตอนเรียกคืนบิล และไม่โผล่บน KDS)
4. **PromptPay QR** — แท็บพร้อมเพย์เป็นแบบ "ยืนยันสลิป" (กรอกเลขอ้างอิง) ยังไม่ generate
   รูป QR (ต้องมี endpoint + ตั้งค่า PromptPay ID ของร้านก่อน)
5. **อีเมลใบเสร็จ** — ปุ่มถูกปิดไว้ (ยังไม่มี mail backend)
6. **ชื่อลูกค้าในรายการบิลที่พัก / กู้คืนลูกค้าตอนเรียกคืนบิล** — เลื่อนไปพร้อมงานส่วนลด
7. **คีย์ลัด `+`/`-`/`Del` รายบรรทัด** — ยังไม่ทำ (ยังไม่มีแนวคิด "บรรทัดที่เลือก")
8. **bump/recall บน KDS** ใช้แตะ + ปุ่ม แทน tap/long-press (long-press ไม่เสถียรข้ามอุปกรณ์)
9. **Spec delta (ตั้งใจ):** ตัวอย่างยอดในตะกร้าคิดค่าบริการตาม *อัตรา* (ตามที่ backend
   คิดจริง) ไม่ได้จำกัดเฉพาะ dine-in อย่างที่ §5.4 เขียน — เพื่อให้ตัวอย่าง = บิลที่ backend สร้าง
10. **shift store** — ใช้ TanStack Query (`useCurrentShift`) แทน Zustand store ตามกฎ
    CLAUDE.md ("data fetching = TanStack Query เท่านั้น")
